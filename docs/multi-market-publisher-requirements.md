# 多市场发包客户端需求文档

## 1. 背景

Andpub 第一阶段要做一个纯本地运行的桌面客户端，支持 macOS、Windows、Linux，用于把 Android 应用提交到多个国内应用市场审核。

第一版必须支持以下 6 个市场：

- 华为 AppGallery Connect
- 荣耀应用市场
- 小米应用市场
- OPPO 应用市场
- vivo 应用市场
- 腾讯应用开放平台

当前 `docs/` 下已有各市场 API 文档，第一版实现以这些文档为准。

## 2. 已确认目标

- 第一版按纯本地客户端实现，不依赖服务端。
- 模块边界预留后续服务端 + Web 形态。
- 第一版覆盖 6 个市场。
- 第一版核心目标是“提交审核”，不是完整上架生命周期管理。
- 发包产物来源同时支持本地文件和公网 URL，通过单选切换。
- 账号密钥先按单人本机使用方案保存。

## 3. 非目标

第一版不优先实现：

- 团队协作、权限、多人审批。
- 云端密钥托管。
- CI 产物自动拉取。
- Web 控制台。
- 完整发布生命周期运营能力，例如全量发布控制、长期灰度运营、拒审后自动重提。
- 应用市场内创建真实应用。客户端内的“创建应用”只是本地建档，市场内应用必须已经存在。

## 4. 核心概念

### 4.1 应用

客户端内维护的应用记录。

客户端必须支持创建和管理多个应用。每个应用以包名作为主要业务标识，应用名只作为展示名称。

必填字段：

- 应用名
- 包名

建议字段：

- 默认隐私政策 URL
- 默认发布备注
- 默认截图、图标、资质文件引用
- 创建时间、更新时间

### 4.2 渠道

渠道表示某个应用关联的具体市场配置。第一版中，渠道等同于应用市场接入实例。

每个渠道需要保存：

- 市场类型
- 市场侧应用标识，例如 appId、APPID 或 open platform id
- 账号密钥配置
- 市场要求的基础资料字段
- 市场能力开关
- 最近一次同步到的应用详情

后续如要支持同一市场多个账号，可以把渠道扩展为：

```text
应用
  └── 渠道
        ├── 市场类型
        ├── 账号配置
        └── 市场侧应用标识
```

### 4.3 发布任务

一次用户点击提交后的发包流程。批量发 6 个市场时，应拆成 6 个独立发布任务。

发布任务需要记录：

- 应用 ID
- 渠道 ID
- 市场类型
- 发包模式：统一产物或按渠道产物
- 产物来源类型：本地文件或 URL
- 包类型：统一 APK、32/64 APK、AAB
- 文件路径或 URL
- 文件 hash
- versionCode、versionName、包名
- 提交状态
- 市场返回的任务 ID、releaseId、requestId、serial number 或 objectId
- 错误码、错误信息
- 请求摘要和响应摘要
- 创建时间、更新时间

## 5. 功能需求

### 5.1 应用管理

用户可以在客户端创建应用记录。

应用列表展示所有本地应用。用户进入单个应用详情后，再管理该应用关联的渠道、市场信息和发布任务。

创建时只要求输入：

- 应用名
- 包名

客户端需要校验包名格式，但不负责在市场内真实创建应用。

### 5.2 渠道管理

用户可以给应用添加 6 类市场渠道。

添加渠道时，客户端需要根据市场类型展示对应配置项：

- 鉴权字段，例如 client_id、client_secret、access_key、access_secret。
- 市场侧应用标识获取方式。优先支持通过包名查询；不支持或查询失败时允许用户手动填写。
- 市场提交审核时必填的基础资料字段。
- 支持的发包形态。

渠道配置保存前需要做最小可用校验：

- 必填密钥不为空。
- 包名匹配当前应用。
- 可以调用应用详情接口的市场，保存前应支持“测试连接”。

### 5.3 应用详情

应用详情页展示应用本地信息和所有渠道状态。

每个渠道至少展示：

- 市场名称
- 连接状态
- 市场侧应用 ID
- 当前线上版本
- 当前审核状态
- 最近一次发布任务状态
- 最近一次失败原因
- 支持能力：统一 APK、32/64 APK、AAB、URL 拉包

能从市场接口获取的详情，按市场能力展示，不强行拉平成完全一致字段。

### 5.4 发包入口

用户在应用详情页发起发布。

发包表单需要包含：

- 目标渠道：可多选。
- 发包模式：统一产物或按渠道产物。
- 发布备注。
- 审核测试账号、密码、说明，按市场能力显示。
- 定时发布时间、分阶段发布等高级字段，第一版可先隐藏或作为可选字段。

发包表单支持两种产物填写模式：

- 统一产物模式：用户填写一套产物信息，提交到多个渠道。适合各市场使用同一个 APK/AAB 或同一个下载 URL 的场景。
- 按渠道产物模式：用户为每个目标渠道分别填写产物信息。适合包内带市场渠道号、不同市场使用不同下载 URL、或某个市场需要特殊包形态的场景。

统一产物模式需要填写：

- 产物来源：本地文件或 URL，单选。
- 包形态：统一 APK、32/64 APK、AAB。
- 本地文件路径或 URL。
- URL 方式所需的 MD5、SHA-256 等 hash。

按渠道产物模式中，每个渠道需要单独填写：

- 产物来源：本地文件或 URL，单选。
- 包形态：统一 APK、32/64 APK、AAB。
- 本地文件路径或 URL。
- URL 方式所需的 MD5、SHA-256 等 hash。
- 渠道专属发布备注或审核资料覆盖项，按市场能力显示。

客户端需要根据目标渠道交集限制可选项：

- 统一产物模式下，如果某市场不支持当前包形态或产物来源，该市场不可选或需要切换到按渠道产物模式。
- 按渠道产物模式下，每个渠道只展示该市场支持的包形态和产物来源。
- 如果某市场支持 URL 拉包但需要 HTTPS，提交前校验 URL。
- 每个渠道最终都会生成一个独立发布任务，发布任务记录该渠道实际使用的产物信息。

### 5.5 本地文件发包

用户选择本地 APK/AAB 文件后，客户端需要：

- 解析包名、versionCode、versionName。
- 计算 MD5、SHA-256，按市场要求使用。
- 校验 APK/AAB 包名和应用包名一致。
- 对 32/64 APK 形态，分别解析并校验两个文件。
- 调用市场上传接口或预签名上传接口。
- 上传成功后调用市场提交审核接口。

### 5.6 URL 发包

用户输入公网 URL 后，客户端需要：

- 校验 URL 格式。
- 对明确要求 HTTPS 的市场强制 HTTPS。
- 允许用户输入 MD5 或 SHA-256。
- 对可访问性做轻量检查。
- 调用市场 URL 拉包接口。
- 拉包接口只上传不提交审核的市场，需要继续调用提交审核接口。

如果 URL 方式无法在本地解析包名和版本，客户端需要明确提示风险，并在市场返回解析结果后更新任务信息。

### 5.7 提交审核

第一版以提交审核成功作为核心闭环。

提交成功的判断不能只看 HTTP 200，需要按市场文档解析业务结果：

- 同步成功：记录为已提交审核。
- 异步受理：记录为处理中，并保存市场任务 ID。
- 失败：记录错误码、错误信息和建议排查项。

### 5.8 状态查询

第一版需要支持手动刷新发布任务状态。

可支持后台轮询，但必须遵守市场限频：

- 荣耀审核结果官方建议低频查询。
- vivo、OPPO 任务状态按任务接口查询。
- 华为异步接口优先支持回调，第一版本地客户端可先使用查询兜底。

### 5.9 日志和诊断

每个发布任务需要保留诊断信息：

- 请求时间线。
- 本地文件名、大小、hash。
- 市场返回的业务码。
- 市场任务 ID。
- 关键请求参数摘要。
- 响应摘要。

密钥、token、签名结果不能明文写入日志。

## 6. 市场能力矩阵

| 市场 | 统一 APK | 32/64 APK | AAB | URL 拉包 | 应用详情 | 提交审核/状态 |
|---|---:|---:|---:|---:|---:|---:|
| 华为 | 支持 | 待确认 | 支持 | 支持 | 支持 | 支持 |
| 荣耀 | 支持 | 待确认 | 未见文档 | 支持 | 支持 | 支持 |
| 小米 | 支持 | 未见文档 | 未见文档 | 未见文档 | 支持 | 支持提交，审核状态以后台为准 |
| OPPO | 支持 | 支持 | 未见文档 | 不支持外部 URL 直填，需先上传到 OPPO | 支持 | 支持 |
| vivo | 支持 | 支持 | 未见文档 | 支持 | 部分支持 | 支持 |
| 腾讯 | 支持 | 支持 | 未见文档 | 未见文档 | 支持 | 支持 |

说明：

- “未见文档”表示当前仓库文档未覆盖，第一版不作为该市场能力开放。
- OPPO 返回资源 URL 后再提交发布，不等同于用户直接传自己的公网 URL。
- 华为 AAB 支持来自当前文档的下载方式和编译状态查询。

## 7. 模块拆分

为后续服务端 + Web 形态预留，第一版按以下边界设计。

```text
composeApp
  ├── ui
  │   ├── app list / app detail
  │   ├── channel config
  │   └── publish wizard
  ├── domain
  │   ├── app model
  │   ├── channel model
  │   ├── artifact model
  │   ├── publish task model
  │   └── market capability model
  ├── application
  │   ├── create app use case
  │   ├── add channel use case
  │   ├── sync app detail use case
  │   ├── prepare artifact use case
  │   └── submit publish task use case
  ├── market
  │   ├── MarketPublisher interface
  │   ├── HuaweiPublisher
  │   ├── HonorPublisher
  │   ├── XiaomiPublisher
  │   ├── OppoPublisher
  │   ├── VivoPublisher
  │   └── TencentPublisher
  ├── storage
  │   ├── local database
  │   ├── secure credential store
  │   └── publish log store
  └── platform
      ├── file picker
      ├── package parser
      └── desktop secure storage
```

服务端化时，优先迁移这些模块：

- `application`
- `market`
- `storage` 中的数据库模型
- 发布任务调度和状态轮询

桌面客户端保留：

- UI
- 本地文件选择
- 本机密钥读取或用户登录服务端

## 8. 本地数据与密钥

第一版建议：

- 普通配置使用本地数据库保存。
- 密钥使用本机安全存储。
- 如果系统安全存储接入成本过高，先使用本地加密文件，但必须把主密码或系统用户能力作为保护边界。

需要避免：

- 密钥写入普通日志。
- 密钥进入导出文件。
- 密钥进入崩溃日志。

## 9. UI 需求

第一版 UI 以工作台形态为主，不做营销页。

主要页面：

- 应用列表
- 创建应用弹窗
- 应用详情
- 渠道配置
- 发布向导
- 发布任务详情
- 设置页

交互要求：

- 表单字段按市场能力动态展示。
- 不支持的能力不可选，并给出市场级原因。
- 批量发布允许部分成功。
- 失败任务可以重新提交。
- 发布前展示确认摘要。

## 10. 待确认项

以下问题不阻塞第一版设计，但实现前需要逐项确认：

- 华为和荣耀是否要支持 32/64 分包。如果没有明确文档，第一版默认关闭。
- 第一版是否要支持定时发布、分阶段发布、灰度发布字段。
- 是否需要导入/导出应用和渠道配置。
- 本地数据库是否需要加密。
- 是否需要发布任务自动轮询，还是只做手动刷新。
- APK/AAB 解析库选型。
- 是否要保留完整原始响应。若保留，需要做脱敏。
