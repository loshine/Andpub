# 多市场发包客户端路线图

## 1. 路线图原则

- 第一版只追求“可靠提交审核”。
- 市场差异通过适配器隔离，不在 UI 和业务流程里散落市场判断。
- 本地客户端先完整跑通，但领域模型、市场适配器、发布编排按后续服务端可复用来拆。
- 每个市场先做最小可用链路，再补高级能力。
- 大文件上传、异步任务、错误码和日志从第一阶段就纳入设计。

## 2. 阶段 0：基础工程整理

目标：把当前示例项目整理成可承载业务的桌面客户端骨架。

任务：

- 建立 UI 导航结构：应用列表、应用详情、渠道配置、发布任务。
- 建立领域模型：应用、渠道、市场能力、产物、发布任务。
- 建立发包模式模型：统一产物、按渠道产物。
- 建立本地存储抽象。
- 建立密钥存储抽象。
- 建立 Ktor HTTP 客户端封装。
- 建立统一错误模型和日志模型。

验收：

- 桌面端可以启动。
- 可以创建本地应用记录。
- 可以新增一个 mock 渠道。
- 可以创建 mock 发布任务并展示状态流转。

## 3. 阶段 1：市场能力建模

目标：把 6 个市场的差异先固化成代码模型，避免后续 UI 和发布流程反复改。

任务：

- 定义 `MarketType`。
- 定义 `MarketCapability`：
  - 是否支持统一 APK。
  - 是否支持 32/64 APK。
  - 是否支持 AAB。
  - 是否支持用户 URL 拉包。
  - 是否支持查询应用详情。
  - 是否支持查询发布状态。
- 定义 `MarketCredentialSchema`。
- 定义 `MarketChannelFormSchema`。
- 定义 `MarketPublisher` 接口。
- 为 6 个市场建立空实现和 fake 实现。

验收：

- UI 可以根据市场能力动态显示发包选项。
- 不支持的组合不能提交。
- fake publisher 可以跑通 6 个市场的任务状态流转。
- 发布表单可以在统一产物和按渠道产物之间切换。

## 4. 阶段 2：渠道配置和应用详情

目标：每个市场都能完成账号配置、连接测试和应用详情查询。

任务：

- 华为：OAuth/token、按包名查 appId、查询应用信息。
- 荣耀：token、按包名查 APPID、查询应用详情。
- 小米：userName/privateKey/publicKey 配置、查询包名状态。
- OPPO：client_id/client_secret、token、查询应用详情。
- vivo：access_key/access_secret、基础签名、任务/应用相关查询能力。
- 腾讯：client_id/client_secret 或文档要求密钥、签名、查询应用详情。
- 渠道配置保存前支持测试连接。
- 应用详情页展示每个渠道的当前状态。

验收：

- 6 个市场都能保存渠道配置。
- 支持查询的市场可以拉取应用详情。
- 查询失败时能显示市场错误码和可读原因。

## 5. 阶段 3：本地产物处理

目标：让发布前校验尽可能在本地完成。

任务：

- 支持选择本地 APK/AAB。
- 支持统一产物模式下的一套产物输入。
- 支持按渠道产物模式下的多渠道独立产物输入。
- 计算 MD5、SHA-1、SHA-256。
- 解析 APK 包名、versionCode、versionName。
- 解析 APK ABI，用于辅助判断 32/64 包。
- 校验包名和应用记录一致。
- 支持 URL 输入和 URL 格式校验。
- 支持用户为 URL 产物输入 hash。

验收：

- 本地 APK 可以展示包信息和 hash。
- 包名不一致时阻止提交。
- 32/64 两包形态可以分别展示信息。
- URL 方式可以进入发布任务，但会提示本地无法完整解析包信息。
- 按渠道产物模式下，每个渠道可以保存并校验自己的文件或 URL。

## 6. 阶段 4：统一 APK 提交审核

目标：6 个市场全部跑通统一 APK 提交审核，这是第一版最关键里程碑。

建议顺序：

1. 小米：链路相对集中，适合先打通签名、multipart、业务错误处理。
2. OPPO：先上传资源再提交，适合验证资源 URL 型流程。
3. vivo：覆盖直传和任务状态。
4. 荣耀：覆盖上传地址、objectId、update-file-info、submit-audit。
5. 华为：覆盖 URL 提交审核和异步状态。
6. 腾讯：覆盖预签名上传、update_app、审核状态查询。

任务：

- 为每个市场实现统一 APK 提交。
- 支持统一产物模式下批量生成多个发布任务。
- 支持按渠道产物模式下按渠道产物生成多个发布任务。
- 统一发布任务状态：
  - created
  - validating
  - uploading
  - submitting
  - accepted
  - submitted
  - failed
- 保存市场侧任务标识。
- 保存脱敏请求摘要和响应摘要。
- 支持失败后重试。

验收：

- 6 个市场都可以从本地 APK 提交到审核或异步受理状态。
- 失败时能看到市场错误码、错误信息和任务日志。
- 批量提交时，一个市场失败不影响其他市场继续执行。

## 7. 阶段 5：URL 发包

目标：对支持用户 URL 拉包的市场开放 URL 方式。

任务：

- 华为：实现 `app-submit-with-file` 或 `app-package-file/by-url` 相关链路。
- 荣耀：实现 `upload-by-url`，并接后续文件绑定和提交审核。
- vivo：实现下载文件方式创建/更新。
- OPPO：不开放用户 URL 直填，只使用 OPPO 上传后返回 URL。
- 小米：当前文档未见 URL 拉包，第一版关闭。
- 腾讯：当前文档未见 URL 拉包，第一版关闭。

验收：

- 选择 URL 来源时，只能选择支持 URL 的市场。
- URL 拉包异步受理后能保存 requestId/taskId。
- 可以手动刷新任务状态。

## 8. 阶段 6：32/64 APK

目标：对明确支持分包的市场开放 32/64 发包。

第一版开放：

- vivo
- OPPO
- 腾讯

暂不开放：

- 华为，除非补充到明确文档。
- 荣耀，除非补充到明确文档。
- 小米，除非补充到明确文档。

任务：

- 发布向导增加 32 位 APK 和 64 位 APK 两个文件位。
- 校验两个包包名一致。
- 校验版本信息一致或符合市场要求。
- vivo 分别走 32/64 上传接口。
- OPPO 使用 `cpu_code` 提交多包信息。
- 腾讯使用 `apk32_flag`、`apk64_flag` 和对应流水号。

验收：

- 支持市场可以提交 32/64 包。
- 不支持市场在 UI 中不可选。
- 两个文件缺失、包名不一致、hash 缺失时阻止提交。

## 9. 阶段 7：AAB

目标：先支持华为 AAB，后续按新增文档扩展其他市场。

任务：

- 发布向导开放 AAB 包形态。
- 仅华为默认支持 AAB。
- 实现华为 AAB 提交。
- 实现华为 AAB 编译状态查询。

验收：

- AAB 形态下只有支持市场可选。
- 华为可以提交 AAB 并查询编译状态。

## 10. 阶段 8：体验和可靠性

目标：把工具从“能跑”打磨到“可日常使用”。

任务：

- 发布前确认页。
- 任务时间线。
- 日志导出。
- 错误码映射和排查建议。
- 手动刷新和低频自动刷新。
- 网络代理设置。
- 上传超时和重试策略。
- 密钥脱敏显示。
- 应用和渠道配置导入导出，默认不导出密钥。

验收：

- 发布失败后能从 UI 看到足够定位的信息。
- 常见市场错误码有可读说明。
- 网络失败、超时、异步处理中都有明确状态。

## 11. 后续服务端化路径

当需要服务端 + Web 形态时，优先按以下方式迁移。

服务端负责：

- 应用、渠道、发布任务数据库。
- 市场账号密钥托管。
- 市场 API 调用。
- 大文件上传中转或对象存储接入。
- 发布任务队列。
- 状态轮询和回调接收。
- 审计日志。

Web 负责：

- 应用管理。
- 渠道配置。
- 发布表单。
- 任务状态展示。
- 日志查看。

桌面客户端可保留：

- 本地 APK/AAB 解析。
- 本地文件选择。
- 上传到服务端或对象存储。
- 单人离线模式。

需要提前保持稳定的接口：

- `MarketPublisher`
- `PublishTask`
- `Artifact`
- `MarketCredential`
- `MarketCapability`

## 12. 推荐 MVP 截止线

第一版可以定义为：

- 可以创建应用。
- 可以添加 6 个市场渠道。
- 可以查询并展示渠道应用详情。
- 可以用本地统一 APK 向 6 个市场提交审核。
- 支持 URL 来源，但只对华为、荣耀、vivo 开放。
- 支持手动刷新任务状态。
- 保留任务日志和错误信息。

32/64 APK 和 AAB 可以作为第一版后半段能力，也可以拆到第二个版本；但代码模型需要从一开始支持，避免后续重构发布表单和任务模型。
